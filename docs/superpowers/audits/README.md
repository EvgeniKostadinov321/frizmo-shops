# Одитни брифове — „тази част на 100%" (2026-07-07)

Четири фокусирани одита след Вълни 1–3Б, за да обявим текущата повърхност
(builder + storefront + checkout + coupons + newsletter) за завършена, преди да
градим нови фийчъри. Всеки бриф е **самостоятелен** и се изпълнява **inline,
последователно** (правило: без паралелни субагенти — пази usage лимита).

## Ред на изпълнение (по риск)

1. [`2026-07-07-audit-1-security.md`](2026-07-07-audit-1-security.md) — multi-tenant изолация, публични endpoint-и, пари, race
2. [`2026-07-07-audit-2-accessibility.md`](2026-07-07-audit-2-accessibility.md) — a11y на storefront + форми + 9-те теми
3. [`2026-07-07-audit-3-performance.md`](2026-07-07-audit-3-performance.md) — Core Web Vitals, image/bundle, кеш/ISR
4. [`2026-07-07-audit-4-ux-conversion.md`](2026-07-07-audit-4-ux-conversion.md) — checkout поток + builder UX + състояния

## Общи правила за ВСЕКИ одит (важат над всичко останало в брифа)

- **Само откриване, без поправки.** Одитът НЕ пипа код. Резултатът е списък с
  находки. Поправките са отделна стъпка след одобрение от потребителя.
- **Стой в scope-а.** Всеки бриф изброява точно кои файлове/папки влизат. Ако
  забележиш нещо извън обхвата си, отбележи го с един ред в „Извън обхват (за друг
  одит)" и продължи — не го разследвай.
- **Не претупвай.** Отвори всеки файл от „Задължителни точки". Не заключавай от
  име на файл; чети кода. Но и не чети целия проект — само изброените места +
  каквото те насочат (grep) към конкретна находка.
- **Икономия на токени.** Ползвай Grep/Glob да стесниш, чети целеви диапазони, не
  цели 2000-редови файлове наведнъж, ако не трябва. Не преразказвай прочетения код
  обратно — само находките.
- **Формат на всяка находка** (в таблица или списък):
  `тежест · файл:ред · какво е · защо е проблем · конкретна препоръка`.
  Тежести: **CRIT** (изпускане/теч/загуба на пари/сигурност) · **HIGH** ·
  **MED** · **LOW/nit**. Ако нищо — кажи го изрично за съответната точка.
- **Обективност.** Твърдиш „бъг" само с път + ред + сценарий на провала. Ако е
  съмнение, маркирай „за проверка", не „бъг".
- **Проектни правила бият.** Чети `CLAUDE.md`, `CLAUDE-backend.md`,
  `CLAUDE-frontend.md`, `AGENTS.md` — находка, която противоречи на съзнателно
  проектно решение (ADR в `docs/decisions/`), не е находка.

Изход на всеки одит: секция „Обобщение (N находки: X CRIT, Y HIGH…)" + таблица,
подредена най-тежкото първо. Накрая: чеклист „проверено ✓ / чисто" по точките.
