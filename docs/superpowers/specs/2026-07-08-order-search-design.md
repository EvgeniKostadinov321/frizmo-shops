# M5 — Търсене на поръчки (спец)

> Втори фийчър от post-audit roadmap-а. Одобрен дизайн 2026-07-08. Стъпва на
> същата orders страница и същия URL-driven патърн като статус филтъра.

## Проблем
Търговец с много поръчки не може да намери конкретна — има само филтър по статус
(`OrderStatusFilter`). Клиент звъни („поръчах вчера") → няма как да я откриеш.

## Решение (обхват)
Поле за търсене над таблицата → филтрира по **номер, име на клиент, телефон**.
Работи заедно със статус филтъра и пагинацията (всичко през searchParams).

**Решения (одобрени):** търсене по номер + име + телефон (не имейл). ILIKE, без
pg_trgm (поръчките на един магазин са малко).

## Архитектура (същия патърн като статус филтъра)

### 1. `getOrders` разширен (`src/db/queries/orders.ts`)
Нов параметър `search?: string`. Изгражда ЕДНО OR-условие (група), добавено с
`AND` към `shopId` + статус (tenant изолация непокътната):
- Винаги: `customerName ILIKE %q%` **OR** `customerPhone ILIKE %digitsOnly(q)%`
  (за телефона търсим само по цифрите на заявката).
- Плюс, само ако заявката е **чисто число** (след махане на `#` и водещи нули):
  **OR** `orderNumber = parseInt(q)`.
- Резултат: `... AND (name ILIKE q OR phone ILIKE digits [OR orderNumber = N])`.
- Празна/whitespace заявка → групата се пропуска (без филтър).
- Ако `digitsOnly(q)` е празно (само букви) → phone клонът се пропуска.

Телефонът се съхранява E.164 (`+359877167172`). За да съвпада „0877"/„877"
срещу него, при търсене махаме нецифровите знаци от заявката и ILIKE-ваме
`%<digits>%` върху `customerPhone`. (Не нормализираме към E.164 — частична
цифра трябва да работи.)

Дължина на заявката ограничена (напр. 60), санитизирана (`sanitizeText`).

### 2. `OrderSearch` — нов клиентски компонент (`src/components/dashboard/`)
Поле за търсене (`<Input>`), навигира с `?q=...` + пази `status` в URL-а.
Debounce ~300ms (да не навигира при всяка буква). URL-driven → пагинация и
споделяне на линк работят. Начална стойност от `searchParams.q`.

### 3. Orders страница (`src/app/(dashboard)/dashboard/orders/page.tsx`)
Чете `searchParams.q`, подава `search` на `getOrders`, рендерира `OrderSearch`
до статус филтъра. При активно търсене страницата се нулира на 1. Пагинационните
линкове носят и `q`, и `status`.

## Edge cases
- Празна заявка → всички поръчки.
- Никакви резултати → съществуващият `EmptyState` (текст „с този критерий").
- SQL injection → Drizzle параметризира ILIKE (безопасно).
- Търсене „#0012" → махаме `#` и водещите нули през parseInt → `orderNumber = 12`.
- Комбинация търсене + статус филтър → и двете важат (AND).

## Извън обхват (YAGNI)
Full-text/pg_trgm · търсене по продукт/сума/дата/имейл · запазени филтри.

## Тестване (потребителят)
1. Направи 2-3 поръчки с различни имена/телефони.
2. Търси по номер („12", „#0012", „0012") → намира точната.
3. Търси по част от име („Евг") → намира.
4. Търси по част от телефон („877") → намира.
5. Комбинирай със статус филтъра → и двете важат.
6. Празно поле → всички се връщат. Несъществуващо → празно състояние.
