# Post-MVP стратегически roadmap

**Дата:** 2026-07-04
**Статус:** посока/план (НЕ изпълнение) — референция за бъдещи спецове
**Обхват:** 6 стратегически теми от реалния опит на собственика. Всяка тема ще получи
собствен детайлен спец + план преди изпълнение. Този документ фиксира ПОСОКАТА,

## §1 — Плащане на абонаменти (търговецът плаща на Frizmo)

### Проблем
Stripe като единствена опция не работи за BG пазара — ниско доверие към „чужди"
платежни системи. Нужни са няколко метода, включително банков път.

### Решение — hybrid, три метода
| Метод | Роля | Автоматизация |
|-------|------|---------------|
| **Банков превод** (IBAN + основание „Frizmo {shopId}") | Най-високо доверие за BG | Хибрид: ръчна реконсилиация сега, авто по-късно |
| **Epay.bg / EasyPay** | Разпознат в BG (каса/ATM/онлайн), за хора без карта/доверие | Полу-авто чрез webhook |
| **Карта (Stripe)** | Международно, удобно за част от търговците | Автоматично |

### Архитектура (ключово решение)
Абстрактен платежен слой от самото начало — интерфейс `PaymentProvider`:
- `initiate(subscription): { redirectUrl?, instructions? }`
- `verifyWebhook(payload): PaymentEvent`
- `getStatus(reference): PaymentStatus`

Имплементации: `BankTransferProvider`, `EpayProvider`, `StripeProvider`. Subscription
логиката не знае за конкретния провайдър — провайдърите само тласкат преходите.

> **Текущо състояние (2026-07-04):** НЯМА `subscriptions` таблица още — `getShopPlan()`
> (src/lib/plan.ts) е stub (всички магазини = "pro", всички в trial). Статусите
> `trial→active→grace→suspended→cancelled` са планираните преходи (План 6 Фаза Б),
> не жива таблица. §1 включва създаването на `subscriptions` + `subscription_payments`.

### Банков превод — реконсилиация (хибрид)
- **Сега:** търговецът вижда IBAN + уникално основание; плаща; в `/admin` ти маркираш
  абонамента като платен (нов admin action). Таблица `subscription_payments`
  (reference, method, amount, status, confirmedBy, confirmedAt).
- **По-късно:** авто-реконсилиация чрез bank API (Revolut Business API или ISO 20022
  statement парсинг) — мачва по основание. Архитектурата го предвижда: същият
  `confirmPayment(reference)` път, извикан от cron вместо ръчно.

### Реалности / рискове
- myPOS/Borica (обмислени, отложени) искат търговски договор + одобрение — не „сложи ключ".
- Epay интеграцията изисква регистрация като търговец в Epay.
- Банковият превод изисква дисциплина за реконсилиация в началото (приемливо при малък обем).

### Сложност: банков превод НИСЪК · epay СРЕДЕН · Stripe СРЕДЕН

## §2 — Куриери (Econt, Speedy)

### Проблем
Търговците въвеждат доставка ръчно. Реалните куриери (Econt, Speedy = двата най-големи)
имат API — интеграцията е голяма стойност.

### Решение — фазово, крайна цел пълно проследяване
Собственикът иска дългосрочно **пълно следене на пратката**, но започваме с най-простото.

| Фаза | Какво | Сложност |
|------|-------|----------|
| **1 (старт)** | **Офис picker** — клиентът избира Econt/Speedy офис от widget (карта/списък) в checkout | Нисък-среден |
| **2** | **Цена на доставка** — при избран офис/адрес API връща реалната цена | Среден |
| **3** | **Товарителница** — търговецът генерира с един бутон (баркод + PDF за печат) | Среден-висок |
| **4 (цел)** | **Проследяване** — статус на пратката се дърпа от куриера, показва се на клиента | Среден |

### Архитектура
Същият pattern като плащанията — интерфейс `CourierProvider`:
- `getOffices(query): Office[]`
- `calculatePrice(from, to, parcel): Price`
- `createLabel(shipment): { trackingNumber, labelPdf }`
- `track(trackingNumber): TrackingStatus`

Имплементации: `EcontProvider`, `SpeedyProvider`. Office picker-ите имат готови JS widget-и.

### Куриерски акаунт
**Всеки търговец със свой** — въвежда своите Econt/Speedy credentials (криптирани,
per-shop, като payment методите). Той има договор с куриера, цените са неговите.
Нова таблица `courier_credentials` (shopId, provider, encryptedCredentials).

### Реалности / рискове
- Econt API: `ee.econt.com` (Shipments, office list). Speedy: `api.speedy.bg`.
- Credentials трябва да са криптирани (не plaintext) — `SUPABASE_SECRET_KEY` ниво.
- Проследяването (фаза 4) зависи от товарителница (фаза 3) — има естествен ред.

### Сложност: фаза 1-2 СРЕДЕН · фаза 3-4 ВИСОК

## §3 — Наличности в реално време

Два ОТДЕЛНИ проблема — коректност и UX.

### §3a — Race condition при поръчка (коректност, БЪГ-В-ЗАРОДИШ)
**Проблем:** двама купуват последната бройка едновременно; и двете поръчки минават.
Текущият `decrementStock` (src/actions/orders.ts) чете `variant.stock` в JS, после
пише `stock - qty` — класически read-modify-write. Ако редовете не са заключени, race.

**Решение — атомарен guard (препоръчвам):**
```sql
UPDATE products SET stock = stock - :qty
WHERE id = :id AND (stock IS NULL OR stock >= :qty)
```
Ако върне 0 засегнати реда → изчерпано → откажи цялата поръчка (rollback транзакцията).
Алтернатива: `SELECT ... FOR UPDATE` на редовете преди декремент (песимистично).

**Приоритет: ВИСОК** — това е коректностен бъг с парично/доверие значение. Малка работа,
голям ефект. Прави се преди реален трафик.

### §3b — Live update на бройката (UX, без ръчен refresh)
**Проблем:** 5 души гледат продукт с 1 бройка; някой купува; бройката трябва да падне
live без F5. Иначе лош UX (поръчваш нещо, което вече е изчерпано).

**Решение — Supabase Realtime (препоръчвам):**
- Storefront компонентът се абонира за промени в `products.stock` / `productVariants.stock`
  през WebSocket (Supabase Realtime — вградено, вече сме на Supabase).
- UI обновява live: „Последни 2 бройки", „Изчерпан" веднага щом падне на 0.
- Алтернативи (отхвърлени): polling (неелегантно), SSE (повече работа).

**Сложност: §3a НИСЪК · §3b НИСЪК-СРЕДЕН**

## §4 — Известия center (камбана + история)

### Проблем
Push е ефимерен (пропуска се при заключен телефон). Нужна е персистентна история —
камбана с брояч, за двете роли (продавач + бъдещ купувач).

### Решение — камбана + история + live
- Нова таблица `notifications` (id, recipientType: `shop|customer`, recipientId, type,
  title, body, data JSONB, readAt, createdAt).
- Камбана иконка в хедъра (dashboard сега; customer портал после) с брояч непрочетени.
- Dropdown/страница със списък + mark-as-read.
- **Push и email стават „транспорти" върху notifications** — създаваш notification запис,
  после по избор го тласкаш push/email. Един източник на истина.
- **Live** чрез Supabase Realtime (същата инфраструктура като §3b) — камбаната „звънва"
  без refresh.

### Типове събития
- **Продавач:** нова поръчка, продукт изчерпан, отказана поръчка, плащане получено,
  абонамент изтича.
- **Купувач** (след §5): поръчка приета, изпратена, доставена, промо в любими.

### Реалности
- Основата (push/email в `src/lib/push.ts`, `email.ts`) я имаме — добавя се персистентен
  слой + UI + Realtime канал.
- Свързва се тясно с §3b (обща Realtime основа) — правят се заедно.

### Сложност: СРЕДЕН

## §5 — Клиентска част (customer акаунти)

### Проблем / текущо състояние
Поръчките са **guest** (`customerName/Phone/Email` като текст в `orders`, без акаунт).
Ок за MVP. Целта: клиент профил със статус на поръчка, доставка, плащания, адреси,
любими, история.

### Решение — guest остава, акаунти като ОПЦИЯ отгоре (фазово)
Guest checkout се пази (най-малко трение). Акаунтите се добавят на етапи:

| Етап | Какво | Стойност / Сложност | Зависи от |
|------|-------|---------------------|-----------|
| **A** | **Проследяване без акаунт** — линк „проследи поръчката" по order № + телефон (вече имаме `/s/[slug]/order/[orderId]`) | Висока / Ниска | — |
| **B** | **Customer акаунт** — регистрация/логин (Supabase роля customer≠merchant), история на поръчки, запазени адреси | Висока / Висока | — |
| **C** | **Любими / wishlist** — добавяне, известие при промо | Средна / Средна | §4 |
| **D** | **Богат портал** — статус на доставка, плащания история | Висока / Висока | §2, §4 |

### Архитектура
- Нови таблици: `customers`, `addresses`, `favorites`; `orders.customerId` (nullable — guest
  поръчките остават с null).
- Нов route-group `(account)` с портал.
- Supabase Auth поддържа множество роли — customer акаунтите са отделни от merchant.

### Реалности / рискове
- Етап D (доставка статус) зависи от §2 (куриери). Богати известия за купувач зависят от §4.
- Това е по същество втори продукт — най-голямата тема, оставя се за накрая.

### Сложност: етап A НИСЪК · B-D ВИСОК

## §6 — Периодично почистване (housekeeping cron)

### Проблем
Данни се трупат без да се чистят — orphan снимки, стари rate_limits, мъртви push
subscriptions, изоставени магазини. Нужни са scheduled функции (ден/седмица/месец).

### Решение — Vercel Cron + housekeeping actions
Vercel Cron (`vercel.json` crons) вика защитени endpoint-и по график. Всяка задача е
чиста функция, тествана изолирано.

| Задача | Какво чисти | Честота (предложение) |
|--------|-------------|----------------------|
| **Осиротели снимки** | Файлове в Storage (`shop-media`), несвързани с продукт/магазин/секция (премахнати при edit, прекъснати качвания). Сравнява Storage списък vs референции в базата. | Седмично |
| **Стари rate_limits** | Редове в `rate_limits` по-стари от 24ч (безполезни след прозореца). | Дневно |
| **Мъртви push subscriptions** | Subscriptions, върнали 404/410 (вече се трият при изпращане — cron гарантира). | Седмично |
| **Изтекли/анулирани магазини** | Магазини без активност след изтекъл trial / анулирани отдавна — архивиране/триене по правила (внимателно: не трий платени данни). | Месечно |

### Архитектура / реалности
- Endpoint-ите са защитени с secret (Vercel Cron header / `CRON_SECRET`), НЕ публични.
- **Осиротели снимки — най-внимателната задача:** триене на грешен файл = загуба на
  клиентска снимка. Изисква сигурна логика (grace период — трий само orphan-и по-стари
  от N дни; dry-run лог преди реално триене).
- Всяка задача логва какво е изтрила (Vercel logs) за одит.
- Изтриване на магазини трябва да уважава legal/данъчни изисквания (поръчки може да
  трябва да се пазят X години) — консултира се преди агресивно триене.

### Сложност: rate_limits/push НИСЪК · снимки СРЕДЕН (внимателен) · магазини СРЕДЕН (правила)

## Зависимости и препоръчан ред

```
§3a Race fix ──────────→ Коректностен бъг. НАЙ-БЪРЗ, направи скоро (малко работа).

§1 Плащания ───────────→ Реален приход + доверие. Независим. Голяма бизнес стойност.
  (банков превод пръв)

§3b Realtime ──┐
               ├─────────→ Обща Supabase Realtime основа. Правят се ЗАЕДНО.
§4 Известия ───┘

§6 Housekeeping ───────→ Независим. Малки задачи първо (rate_limits/push), снимки внимателно.

§2 Куриери ────────────→ Фазово (office picker → tracking). Голяма стойност за търговци.

§5 Customer ───────────→ Най-голямо. Етап A (проследяване) бърз и независим.
  A → B → C(§4) → D(§2,§4)   B-D голям проект, зависят от §2 и §4. Накрая.
```

### Препоръчан приоритет (собственикът избира финално)
1. **§3a race fix** — коректностен бъг, малка работа.
2. **§1 плащания** (банков превод старт) — приход + доверие.
3. **§3b + §4** заедно — обща Realtime основа, голям UX скок.
4. **§6 housekeeping** — паралелно, малки задачи.
5. **§2 куриери** (office picker) — стойност за търговци.
6. **§5 customer** — на етапи, започвайки с проследяване (A).
