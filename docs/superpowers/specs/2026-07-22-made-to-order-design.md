# Ръчна изработка / По поръчка (Made-to-Order) — дизайн

> Спец от брейнсторм 2026-07-22. Реален gap за ръкоделческата публика (картини, шиене,
> бижута) — потвърден от реални бъдещи потребители (майка на основателя + приятелка).
> Pure-code функция, нула външни ключове. Категория 1.

## Проблем

Ръкоделец продава **и готови бройки, И по поръчка** — за един и същ продукт. Пример:
5 готови ръчно плетени гривни. Като свършат, майсторът може да изработи още — но това
отнема време. Днес платформата има само класически `stock`: при `stock=0` продуктът е
„изчерпан" и checkout се блокира. Няма как да се каже „свършиха готовите, но приемам
поръчки по изработка, очаквайте 10–14 дни".

## Решение (в едно изречение)

Per-продукт флаг „приема поръчки по изработка при изчерпване" превръща `stock=0` от
задънена улица („изчерпано") в **режим по поръчка** — купувачът пак поръчва, вижда срок
за изработка и горд badge „Ръчна изработка", а поръчката се маркира като „по изработка".

## Модел на поведение — един продукт, три състояния

Продуктът плавно преминава между състояния според наличността:

| Състояние | Условие | Купувачът вижда | Checkout |
|---|---|---|---|
| **Готово** | `stock > 0` | „N в наличност" (както сега) · изпраща веднага | ✅ декрементира stock |
| **По поръчка** | `stock = 0` + `madeToOrder` + под тавана (или без таван) | badge „Ръчна изработка" + „Изработва се специално за теб · 10–14 дни" | ✅ приема, **без** декремент под 0 |
| **Спряно** | `stock = 0` + (НЕ `madeToOrder` ИЛИ таванът достигнат) | „Изчерпано" / „Опашката е пълна" | ❌ блокира (както сега) |

**Ключово прозрение:** нормалният `stock` механизъм не се пипа. Добавя се само „escape
hatch" при `stock=0` за made-to-order продукти. `stock IS NULL` (не следи наличност) си
остава отделно поведение — минава винаги, без made-to-order логика.

## Нови полета (products таблица + Zod схема)

```
madeToOrder      boolean  NOT NULL DEFAULT false   -- приема поръчки по изработка при stock=0
leadDaysMin      integer                           -- срок изработка, долна граница (напр. 10)
leadDaysMax      integer                           -- срок изработка, горна граница (напр. 14)
madeToOrderCap   integer                           -- таван на опашката по поръчка; NULL = неограничено
```

**Валидация (Zod):**
- `madeToOrder` булев тогъл.
- Когато `madeToOrder = true`: `leadDaysMin` и `leadDaysMax` са **задължителни**, цели,
  `1 ≤ min ≤ max ≤ 365`. Когато `false`: и двете null/празни (не се показват).
- `madeToOrderCap`: незадължителен, цяло `≥ 1` или празно (= неограничено). Само релевантен
  при `madeToOrder = true`.
- Валидацията е cross-field (`superRefine`) — min ≤ max; lead дни задължителни ако тогълът е вкл.

## „По поръчка" наличност — таван на опашката

Когато `stock = 0` и `madeToOrder`, купувачът приема поръчки по изработка. Опционален таван
(`madeToOrderCap`) ограничава едновременните поръчки по изработка, за да не се презапише
майсторът за месеци напред.

**Как броим „активни поръчки по изработка" (без ново броячно поле):**
Заявка към `order_items` join `orders`: редове за този продукт, които са били поръчани в
made-to-order режим (маркер в snapshot-а, виж по-долу) И чиято поръчка е в незавършено
състояние (`new` / `confirmed` / `shipped` — НЕ `completed` / `cancelled`). Броят им спрямо
`madeToOrderCap` решава дали приема още.

**Cap проверка на checkout:** ако `stock=0` + `madeToOrder` + `cap != null`, преброй активните
поръчки по изработка ПРЕЗ транзакцията (под лока) → ако `count >= cap` → блокирай реда с
съобщение „Опашката за изработка е пълна в момента. Върни се по-късно."

## Промяна в поръчковия поток (най-деликатната част)

`decrementStock` (src/actions/orders.ts:57) прави атомарен `stock >= qty` guard в транзакция.
Днешното поведение при `stock < qty` → редът се отхвърля (изчерпано). Промяна:

**Нова логика per ред в транзакцията (запазва race-safety):**
1. Опит за нормален декремент (`stock >= qty` guard, RETURNING) — както сега.
2. Ако успее → готова бройка, редът е „готов". Продължи.
3. Ако НЕ успее (недостатъчна наличност):
   - Ако продуктът НЕ е `madeToOrder` → **отхвърли реда** (изчерпано, както сега).
   - Ако е `madeToOrder`:
     - Провери тавана (заявка под лока, виж горе). Ако достигнат → **отхвърли** („опашката пълна").
     - Иначе → **приеми реда като „по изработка"**, БЕЗ да декрементираш stock (той е вече 0;
       готовите свършиха, това е нова изработка). Маркирай реда `madeToOrder=true` в snapshot.

⚠️ **Concurrency:** `SELECT ... FOR UPDATE` върху продуктовия ред + cap заявката в същата
транзакция → два едновременни опита за последното capacity място → вторият вижда актуалния
брой под лока → точно cap се спазва (както overselling guard-ът сега). Verify скрипт добавка.

⚠️ **Смесени количества:** ако купувач поръча 3 бр, а има 1 готова → или (а) всичко минава
като „по изработка" (по-просто), или (б) 1 готова + 2 по изработка (сложен UX). **Решение за
простота: ако поръчаното количество > наличния готов stock → ЦЯЛОТО количество е „по изработка"**
(не смесваме един ред). Документира се в UI: „Част от бройките са готови, но при поръчка над
наличното цялата поръчка се изработва."

**Следствие за показването:** режимът „по изработка" зависи от **конкретната поръчка спрямо
наличността в момента на поръчване**, не само от `stock=0`. На продуктовата страница показваме
готовата наличност + badge „Ръчна изработка" (гордост, ако продуктът е handmade). Дали
конкретна покупка става „по изработка" се решава на checkout (когато знаем количеството). Ако
`stock=0` изцяло → продуктовата страница директно показва акцентната „по поръчка" кутийка.
Ако `stock>0` → показва наличност, а „по изработка" уточнението идва в кошницата при количество
над наличното.

## Snapshot в order_items

Добавяме към `order_items`:
```
madeToOrder     boolean NOT NULL DEFAULT false   -- редът е поръчан по изработка
leadDaysMin     integer                          -- snapshot на срока към момента
leadDaysMax     integer
```
Причина: order_items вече snapshot-ва име/цена/вариант към момента. Made-to-order статусът и
срокът трябва да оцелеят, дори продуктът после да се промени. Ползва се за: dashboard изглед,
потвърждение към купувача, броене за тавана.

## UX / показване (дизайн език „Пазарен ден")

> Позициониране: **горд знак за качество**, не извинение. Ръкоделците се гордеят с ръчния
> труд. „Изработва се специално за теб" > „съжаляваме, бавно е". Дребен шрифт-извинение е
> забранен. Минава през `docs/design-final-guide/` при имплементация (правило на проекта).

**Продуктова страница (storefront):**
- Badge „Ръчна изработка" (нов, консистентен с „Нов"/промо баджовете — `product-badges` паттерн,
  storefront `--sf-*` токени).
- Когато е в режим по поръчка (stock=0): акцентна кутийка (не дребен ред) — „Изработва се
  специално за теб · очаквайте 10–14 дни". Позитивен тон.
- Когато има готови (stock>0) но продуктът Е handmade: badge „Ръчна изработка" се показва
  (гордост), но доставката е нормална („N в наличност").

**Продуктова карта (каталог/списък):** малък „Ръчна изработка" badge (дискретен, ъгъл на картата
както „Нов").

**Кошница + checkout:** ред „⏳ Изработка 10–14 дни" до продукта, за да е ясно преди плащане.
Ако количката смесва готови + made-to-order продукти → показва най-дългия срок като общ
ориентир („част от поръчката се изработва до X дни").

**Потвърждение на поръчката (storefront):** ако редът е по изработка → „Изработва се специално
за теб · очаквай след ~14 дни", вместо „изпращаме днес".

**Dashboard (търговец):** поръчката/редът показва „по изработка · срок 10–14 дни" бадж, за да
не обърка made-to-order с готова бройка. По желание: филтър „по изработка" в поръчките.

**Продуктова форма (търговец):** нова секция (в „Детайлно" изгледа на тогъла) — „Ръчна изработка
/ по поръчка": тогъл + при вкл. → срок (min–max дни) + опционален таван на опашката. InfoHint
пояснения („Приемай поръчки дори когато готовите бройки свършат"). Мобилно-първо.

## Обхват — какво Е и какво НЕ е в тази функция

**Е:**
- Per-продукт made-to-order флаг + срок (min–max) + опционален таван.
- Плавен преход готово → по поръчка при stock=0.
- Race-safe checkout с cap enforcement.
- Snapshot в order_items + показване навсякъде по потока (продукт, карта, кошница, checkout,
  потвърждение, dashboard).
- Badge „Ръчна изработка" (гордост, показва се и при налични готови бройки).

**НЕ е (YAGNI):**
- Опашкова система с приоритети/пренареждане.
- Автоматични known срокове от куриер (това е майсторов срок, не доставка).
- Частично смесване в един ред (1 готова + 2 изработка) — цялото количество е по изработка при
  недостиг (простота).
- Депозит/предплата за изработка (отделна тема; плащането си е както сега).
- Известия „изработката започна/готова" (по-късно, ако потрябва — стъпва на статус имейлите).

## Тестове (задължителни)

- Zod: madeToOrder тогъл, cross-field (lead дни задължителни при вкл.; min≤max; cap≥1 или празно).
- Pricing/checkout unit: stock>0 → готово; stock=0+madeToOrder → приема без декремент;
  stock=0+НЕ madeToOrder → блокира; cap достигнат → блокира.
- Concurrency (verify скрипт добавка): 2 паралелни поръчки за последното cap място → точно 1 минава.
- Snapshot: order_items пази madeToOrder + срока към момента, оцелява при промяна на продукта.

## Отворени въпроси за плана

- Точен праг „смесено количество" (потвърдено: цялото количество → по изработка при недостиг).
- Дали cap заявката да е материализирана (кеширан брояч) при много поръчки — засега директна
  заявка под лока (простота; оптимизация само ако стане тясно място).
