> Одобрен дизайн 2026-07-08. Ред на имплементация: **S5 → S4**, всеки с гейт
> (`pnpm check`) + commit. Потвърдено от потребителя: (1) S4 body = обикновен
> текст (без rich editor/снимки); (2) S5 БЕЗ конверсия (не следим трафик по

## S5 — Аналитика с тренд

### Решение (нула DB миграция — смята се от orders/order_items/subscribers)
Нова страница `/dashboard/analytics` + линк „Аналитика" в dashboard навигацията.

1. **Период:** 7 / 30 / 90 дни, URL-driven (`?period=30`, дефолт 30). Само
   тези три стойности; друго → дефолт.
2. **Метрики за периода** (изключва `cancelled`, конвенцията на
   `getMonthRevenue`): приходи (сума `total_cents`) · брой поръчки · средна
   поръчка (приходи/брой, 0 при нула поръчки) · нови абонати (confirmed за
   периода). Всяка със сравнение спрямо ПРЕДИШНИЯ период със същата дължина
   („+18%"; при предишен = 0 → „—" без процент).
3. **Графика приходи по дни:** чист SVG bar chart, БЕЗ външна библиотека.
   Дни без поръчки = нулеви колони; `<title>` per колона (дата + сума) за
   достъпност; при 90 дни колоните са тънки, но остават по дни.
4. **Топ 5 продукта** по приход за периода — от `order_items` snapshot-ите
   (`product_name`, работи и за изтрити продукти), group by име: приход + брой
   продадени. Една SQL заявка.
5. **Query слой:** нов `src/db/queries/analytics.ts` — `getAnalytics(shopId,
   periodDays)` връща всичко за страницата (агрегати + дневни редове + топ
   продукти) с минимален брой заявки (3–4, паралелни).

## S4 — Изпращане на newsletter кампании

### Нова таблица `campaigns`
`id, shopId (FK cascade), subject, body, recipientCount, createdAt` + индекс
shopId. `.enableRLS()`. История + одит „какво е пратено на кого/колко".

### Решение
1. **UI в `/dashboard/subscribers`:** секция „Нова кампания" — тема (≤120) +
   текст (textarea, ≤5000, обикновен текст с параграфи). Показва „ще се
   изпрати до N потвърдени абонати"; ConfirmDialog преди пращане. Отдолу:
   история на кампаниите (тема · дата · брой получатели).
2. **Action `sendCampaign`** (`src/actions/newsletter.ts`): през
   `requireShop()`; Zod + санитизация; взима САМО `confirmed` абонати; при 0
   абонати → грешка. Праща с `Promise.allSettled` (последователно стартирани,
   без опашка — ОК до няколкостотин; опашка при мащаб = извън обхват).
   Записва кампанията с реалния брой успешно изпратени.
3. **Имейл шаблон** (`src/lib/email.ts`): `shell()` с името на магазина като
   заглавие, параграфите от body (esc + split по празен ред), и ЗАДЪЛЖИТЕЛЕН
   footer линк „Отпиши се" през съществуващия token механизъм
   (`/s/{slug}/newsletter/confirm?token=X&action=unsubscribe`).
4. **Защита:** лимит 1 кампания/час per магазин през `rate_limits`
   (`campaign:{shopId}`) — срещу двойно цъкане/спам.

## Извън обхват (YAGNI)
Rich text/HTML editor · снимки в кампании · сегментиране на абонати ·
насрочено изпращане · A/B · опашка за масово изпращане · open/click tracking ·
конверсия в аналитиката (иска трафик данни).

## Тестване (потребителят)
- **S5:** /dashboard/analytics — смени 7/30/90 → метрики/графика/топ продукти
  се променят; сравнението показва посока; магазин без поръчки → нули без
  грешки.
- **S4:** напиши кампания → потвърди → имейлът пристига на потвърден абонат с
  работещ „Отпиши се" линк; отписаният НЕ получава следващата; историята
  показва кампанията; втора кампания в същия час → отказана с ясно съобщение.
