# Onboarding / Setup Wizard — продуктов и UX дизайн

**Дата:** 2026-07-05
**Статус:** одобрена посока (спец) — НЕ изпълнение
**Обхват:** пътят на нов търговец от създаден магазин до готов, приличен сайт
**Тип:** продуктово/UX планиране; всяка под-тема получава собствен план преди код

## Проблемът (от опита на собственика)

1. **Сайтовете, които произвеждаме, не са достатъчно добри.** Изходът зависи
   изцяло от входа на потребителя (снимки, тексове, ред на секции, цветове), а
   средният търговец не знае как да построи добър магазин. Дори с перфектен
   редактор, слаб вход → слаб изход.
2. **Твърде сложно е.** След създаване на магазина потребителят е хвърлен в празен
   dashboard/редактор с всичко наведнъж. Няма насока, няма фокус.

## Философия (какво решаваме)

Двата проблема се решават с **един** механизъм: setup wizard, който **сглобява
готов, приличен сайт** вместо да оставя потребителя пред празен редактор.

> **Wizard-ът не иска перфектност. Иска „не-празен, красив старт за ~3 минути".
> Донастройката е в редактора, с постоянно послание „всичко се променя по-късно".**

Ключово предимство спрямо конкурентите: типът магазин (`businessCategory`) вече е
избран при създаването на магазина (`ShopWizard`). Wizard-ът стартира с готов
контекст и **не пита пак за тип** — направо предлага подходящи концепции.

## Решени продуктови развилки (от брейнсторминга 2026-07-05)

| Развилка | Решение |
|---|---|
| Обхват на wizard-а | **Хибрид**: 3 задължителни стъпки (ядро) → готов сайт → 3 опционални бонус стъпки |
| Медия по категория | **Красиви placeholder-и/градиенти** засега (нула asset/лицензионен проблем); истинските снимки идват от търговеца. AI библиотека = бъдеща опция |
| Dark storefront | **Тъмните теми са ОТДЕЛНИ теми** в каталога (не light/dark превключвател на всяка тема). Търговец избира „Оникс" (тъмна) или „Глина" (топла светла) — темата носи цялото си усещане. Нула превключватели, нула двойно тестване |
| Продукти в wizard-а | **1 продукт** (име+цена+снимка) в ядрото — пълни featured секцията, свързва се с текущия „първи продукт" onboarding |
| Preview на тема | **Мини hero preview на живо** — всяка тема е карта с реален hero (заглавие+бутон+цветове+шрифт) с примерно съдържание по категорията |

## Позициониране в съществуващата верига

Днес: `DashboardWelcome` → `/dashboard/onboarding` (`ShopWizard`: име/категория/
контакти/работно време) → празен dashboard. Езикът „всичко може да се променя
по-късно" вече съществува в welcome екрана — wizard-ът го продължава.

Нов поток: `ShopWizard` (създава магазина) → **[НОВ setup wizard]** → dashboard/
редактор. Прогресът се пази в `site_settings.draft` (колоната вече съществува —
нула нова инфраструктура; същият draft, който live preview-то ползва).

## Структура

### Част 0 — Покана (intro екран)

Един екран преди wizard-а: „Ще ти помогнем да построиш професионален сайт за
няколко минути" + „По-удобно е от компютър" + бутони:
- **„Да, помогни ми"** → стартира wizard-а
- **„Ще го направя сам"** → директно в редактора (escape hatch)

### Ядро — 3 задължителни стъпки

**Стъпка 1 — Концепция / Тема**
- Въпрос: „Как искаш да изглежда магазинът ти?"
- Показва 2–3 **препоръчани** теми за категорията, всяка карта с **мини hero
  preview на живо** (заглавие+бутон+цветове+шрифт, примерно съдържание по
  категорията). Под всяка: за какъв бизнес е, какво усещане, защо я препоръчваме.
- Линк „Виж всички теми" → пълен каталог (вкл. тъмните теми).
- Резултат: `theme`, `primaryColor`, `accentColor` в draft.

**Стъпка 2 — Бранд (лого + акцент)**
- Въпрос: „Добави логото и цвета на бранда си."
- Качване на лого (по избор — fallback с инициал вече съществува) + фина корекция
  на палитрата. **НЕ** пита за шрифт/размер/light-dark отделно — темата ги носи.
- Постоянно послание: „Всичко се променя по-късно."
- Резултат: `logoPath` (+ евентуална корекция на цветовете).

**Стъпка 3 — Първи продукт**
- Въпрос: „Добави първия си продукт — така магазинът ти оживява."
- Олекотена форма (име + цена + 1 снимка) → пълни `featured-products` секцията.
- Свързва се с текущия „първи продукт" onboarding, не го дублира.
- Резултат: 1 продукт създаден.

### Момент на „уау" (финал на ядрото)

Full-screen: **„Готово! Ето твоят магазин."** — показва сглобения сайт на живо
(hero + продукта + подходящи секции по категорията). Три пътя:
- **„Публикувай сега"** → онлайн
- **„Искаш още 5 минути за по-добър сайт?"** → бонус стъпките
- **„Ще донастроя сам"** → редактора

Тук се случва магията: wizard-ът е сглобил **стартов набор секции по категорията**
(не празния default hero+featured+contact). Примерни набори:
- Козметика → hero + featured + trust-badges + testimonials
- Ръчна изработка → hero + image-text (историята) + gallery + featured
- Дрехи/мода → hero (slideshow) + featured + gallery + category-grid
- Електроника → hero (split) + featured + trust-badges + faq
- „Друго" → универсален (hero + featured + contact-map)

### Бонус — 3 опционални стъпки (само ако продължи)

**Стъпка 4 — Доверие:** trust-badges + кратък „за нас" текст (историята — силно за
ръчна изработка/храни).
**Стъпка 5 — Още съдържание:** според категорията — галерия (мода/изделия), FAQ
(техника/козметика) или отзиви. Показваме само релевантните.
**Стъпка 6 — Доставка и плащане:** мост към `fulfillment` (наложен платеж,
доставка) — за да е магазинът оперативен, не само красив.

## Edge cases

| Случай | Решение |
|---|---|
| Затвори по средата | Draft записан → следващо влизане „Продължи оттам?" |
| Пропусне всичко | Красиви дефолти по категория (не празно) |
| „Друго" категория | Fallback към универсален набор теми (classic/modern) |
| Няма лого | Fallback с инициал (вече в storefront header-а) |
| Няма снимки | Placeholder-и/градиенти по темата (не счупен layout) |
| Мобилно | Работи, но с напомняне „по-добре на компютър" (както intro модала) |
| Вече минал wizard-а | Не се показва пак (флаг); достъпен ръчно от редактора |

## Зависимости (предусловия ПРЕДИ самия wizard)

Тези две са предпоставки и вероятно стават **отделни планове/стъпки** преди wizard UI-а:

1. **Разширяване на темите.** Днес има 3 (`classic`/`modern`/`warm`, всички светли,
   в `src/lib/themes.ts`). Нужно:
   - тъмни теми като отделни записи (нови `--sf-*` стойности, тъмен фон/текст);
   - повече светли теми за разнообразие;
   - мапинг **категория → 2–3 препоръчани теми**;
   - метаданни на всяка тема (за кого е, какво усещане, защо — за preview картите).
   - ⚠️ Всяка нова тема се тества през 13-те секции (визуален pass).

2. **Стартови набори секции по категория (templates).** Мапинг
   `businessCategory → Section[]` (заместващ универсалния `defaultSections()` от
   `src/lib/sections.ts` при минаване през wizard-а). С примерни заглавия/тексове,
   които изглеждат добре празни или леко попълнени.

## Извън обхвата (за по-късно)

- AI-генерирана медия библиотека по категория (Magnific) — засега placeholder-и.
- Inline редакция върху preview-то.
- Промяна на storefront схемата или мутациите извън добавяне на theme записи.
- L10n/мулти-език.

## Как решава двата проблема

1. **„Сайтовете не са добри"** → темите носят усещането в пакет; стартовият набор
   секции е куриран по категория; „уау" екранът задава очакване за качество; тъмни
   теми за бизнеси, които ги изискват.
2. **„Твърде сложно"** → 3 леки стъпки до готов сайт; всичко останало опционално;
   постоянно „променя се по-късно"; един избор (тема) замества пет решения
   (шрифт/цвят/радиус/усещане/светло-тъмно).
