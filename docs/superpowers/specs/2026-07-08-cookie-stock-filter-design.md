## M4 — Cookie consent (правна коректност)

### Проблем
BG/EU законът иска opt-in за non-essential cookies (глоба 250–2000 лв). Сегашният
банер (`src/components/ui/cookie-consent.tsx`) е информативен „Разбрах".

### Решение
Проектът НЕ ползва проследяващи/рекламни cookies — само сесия (строго необходими).
Opt-in важи САМО за non-essential, каквито няма → без категории/toggle-и (не би
имало какво да се opt-in-ва). Прецизираме само текста:
- Бутон „Разбрах" → **„Приемам"**.
- Маха се `🍪` емоджито (правилото на проекта забранява емоджи в UI).
- Текстът остава: „само строго необходими бисквитки — за вход и сесия. Без
  проследяване, без реклами." + линк към /privacy.

**Файл:** само `cookie-consent.tsx`. Нула DB, нула нова инфраструктура.
**Бележка за бъдеще:** щом дойде аналитика (S5) / pixel → тогава пълен opt-in с
категории. Сега би бил overkill.

## S12 — Складови аларми (в панела)

### Проблем
Наличностите се следят (`products.stock`, `productVariants.stock`), но търговецът
няма индикация при нисък/нулев склад — научава чак когато поръчка се провали.

### Решение (визуална аларма в dashboard, без имейл/push спам)
Праг: `LOW_STOCK_THRESHOLD = 3`.
- `stock === 0` → **изчерпан**; `0 < stock <= 3` → **нисък**. `stock === null`
  (не следи наличност) → игнориран. Важи и за продукт, и за вариант.

**Части:**
1. **Query** `src/db/queries/products.ts`: нова `countLowStock(shopId)` — брой
   активни продукти с ниска/нулева наличност. **Опростяване:** броим по
   ПРОДУКТОВАТА наличност (`products.stock` не null и `<= 3`) — това е една
   заявка без join. Вариантната наличност се пренебрегва за брояча (продуктите с
   варианти обикновено имат `products.stock = null`; тяхната аларма е втора вълна,
   ако потрябва). Същият критерий и за филтъра `stock=low` — консистентно.
2. **Табло** `src/app/(dashboard)/dashboard/page.tsx`: нов `StatTile` „Нисък
   склад" (брой) → линк към `/dashboard/products?stock=low`. `accent` при > 0.
3. **Продуктов списък** `getProducts` + страницата: опционален филтър
   `stock=low` (в наличност ≤3 или 0) + визуален маркер на реда („Нисък склад" /
   „Изчерпан" badge). URL-driven.

Нула DB миграция (`stock` вече съществува).

## S6 — Ценови + наличност филтър (двата листинга)

### Проблем
Каталогът (`/products`) и storefront (`/s/{slug}/products`) имат search/category/
sort, но няма ценови диапазон и „само в наличност".

### Решение
Двете query функции получават нови филтри:
- `searchCatalogProducts` (`src/db/queries/catalog.ts`) — платформен каталог.
- `getActiveProducts` (`src/db/queries/storefront.ts`) — storefront магазин.

Нови параметри: `minPrice?: number` (центове), `maxPrice?: number`, `inStock?: boolean`.
- **Цена:** по ЕФЕКТИВНАТА цена = `coalesce(promo_price_cents, price_cents)`
  (консистентно със сортирането, което вече ползва това в каталога).
  `min` → `effective >= min`; `max` → `effective <= max`.
- **Наличност:** `inStock` → `stock IS NULL OR stock > 0` (null = не следи склад
  → винаги наличен). *(Вариантната наличност НЕ се проверява тук — на ниво
  листинг е достатъчна продуктовата; вариантът се избира на продуктовата страница.)*

**UI:** нов малък компонент (напр. `PriceStockFilter`) — две ценови полета (от–до)
+ чекбокс „В наличност", URL-driven (`?min=&max=&inStock=1`). Преизползван на двата
листинга. Входовете се парсват в центове (`toCents`), невалидни се игнорират.
Пагинационните линкове носят новите параметри.

  филтрите.
