# Варианти на секциите — дизайн

**Дата:** 2026-07-05
**Статус:** одобрена посока (спец) — НЕ изпълнение
**Обхват:** всяка storefront секция получава 2 layout варианта (основен + алтернативен)
**Ред:** СЛЕД темите (спец 2026-07-05 storefront-themes-catalog), преди/паралелно с
стартовите набори секции
**Зависимост за:** setup wizard (по-богато разнообразие на генерираните сайтове)

## Мотивация

Днес всяка секция има ЕДИН фиксиран layout (напр. featured-products винаги
`grid-cols-2 md:grid-cols-4`). Двама търговци с една тема получават почти еднакви
секции. Темата сменя цвят/шрифт/усещане, но не и **подредбата**.

Добавяме на всяка секция **2 варианта на изглеждане** (основен + един алтернативен).
Комбинацията **тема × вариант × ред × съдържание** дава реално разнообразие — двама с
една тема правят коренно различни магазини. Това е и стандартът при Shopify/Wix
(секции с layout варианти).

hero секцията вече има зародиш на това (`layout: full | split | slideshow`) — тук го
правим консистентно правило за всички секции.

## Ключово решение (от брейнсторминга 2026-07-05)

- **Точно 2 варианта на секция** (основен + алтернативен). Не повече — консистентно,
  предвидимо, не претоварва нетехническия търговец, а работата е известна (13 секции ×
  1 нов вариант).
- **Избор чрез dropdown** в редактора на секцията (SectionForm), точно както hero
  layout сега — „Изглеждане: Основно / [Алтернативно име]". Не визуален picker (по-малко
  работа, без нужда от preview асети).
- hero остава изключение с 3 варианта (вече ги има) — не ги режем.

## Данни

Ново поле `variant` в **всяка** секция (в `base` на schema-та):

```ts
const base = {
  id: z.uuid(),
  enabled: z.boolean().default(true),
  variant: z.enum(["primary", "alt"]).default("primary"),
};
```

`.default("primary")` → всяка секция без изрично поле получава „primary" при parse
(нула миграция; толерантният parse така или иначе пази). hero **не** ползва това поле —
той пази своя `layout` в `data` (не пипаме съществуващото).

Storefront рендерът чете `section.variant` и избира layout. `renderSection` подава
`variant` към компонента (или компонентът го чете от `section`).

## Двата варианта per секция (спецификация)

| Секция | Основен (primary) | Алтернативен (alt) |
|---|---|---|
| **hero** | (запазва 3-те: full/split/slideshow) | — без промяна |
| **announcement** | лента цял ред | центрирана, по-компактна |
| **featured-products** | грид 4 колони | карусел (хоризонтален скрол) |
| **category-grid** | грид карти | компактни chip-ове/списък |
| **promo-banner** | снимка отдясно + текст | пълноширок фон + текст върху него |
| **image-text** | снимка отстрани (по избор ляво/дясно, вече в data) | снимка отгоре, текст отдолу (стек) |
| **rich-text** | ляво подравнен | центриран, тесен (editorial) |
| **testimonials** | грид карти | голям единичен цитат / карусел |
| **trust-badges** | хоризонтална лента | грид 2×3 с по-едри икони |
| **gallery** | равна решетка | masonry (разновисоки) |
| **faq** | акордеон единична колона | две колони |
| **contact-map** | карта + инфо един до друг | инфо отгоре, карта пълноширока отдолу |
| **socials** | центрирани икони | лента с етикети |

Забележка: alt вариантите са РЕАЛНИ структурни разлики (не само отстояния), но всеки е
управляема добавка към съществуващия компонент (условен клас/подредба по `variant`).
Всички четат само `--sf-*` (темонезависими).

## Технически детайли (какво се пипа)

- `src/schemas/site-settings.ts`: `variant` в `base` (важи за всички секции освен че
  hero го игнорира). Типът `Section` автоматично го получава.
- `src/lib/sections.ts`: `newSection()` вече включва `variant: "primary"` (от default).
- Секционните компоненти в `src/components/storefront/sections/*`: всеки чете `variant`
  и рендерира alt подредбата при `"alt"` (условно). Малки, изолирани промени.
- `src/components/storefront/sections/index.tsx` (`renderSection`): подава `variant`
  (или компонентът го чете от подадената секция — предпочитано: подаваме цялата секция
  или `variant` изрично).
- `src/components/dashboard/website/section-form.tsx`: Select „Изглеждане" в началото
  на всяка секция (освен hero, който вече има свой layout Select). Етикети на BG,
  специфични за секцията (напр. featured → „Грид / Карусел").
- Storefront: **темите не се пипат**; вариантите са ортогонални на темите.

## Edge cases

| Случай | Решение |
|---|---|
| Секция без `variant` | default „primary" (schema) |
| Невалиден `variant` | толерантният parse → секцията пада към валидна или се изпуска; на практика enum-ът гарантира само primary/alt |
| alt вариант с малко съдържание | трябва да изглежда добре и празен (както primary) — визуален pass |
| hero | не ползва `variant` — 3-те му layout-а остават в `data.layout` |

## Извън обхвата

- 3-ти+ вариант на секция (само 2, освен hero).
- Визуален picker с preview (dropdown е достатъчен).
- Per-вариант допълнителни настройки (напр. брой колони като свободно число) —
  вариантът е фиксиран пресет.
- Промяна на темите или на `data` схемите на секциите (само добавяме `variant` в base).

## Тестове

- Unit: schema parse-ва секция с/без `variant` (default primary); невалиден variant
  се отхвърля; `newSection` връща `variant: "primary"`.
- Визуален pass (собственик): всяка секция × 2 варианта × поне 2 теми (1 светла + 1
  тъмна), light + 375px. Празен alt вариант изглежда добре.
- `pnpm check` зелено.

## Връзка с останалите зависимости

- **Темите** (преди това) — вариантите са ортогонални; работят с всяка тема.
- **Стартови набори секции** (след/паралелно) — наборите могат да задават не само кои
  секции, но и кой `variant` (напр. мода template → featured „карусел"; занаяти →
  gallery „masonry"). Финализира се в онзи спец.
- **Setup wizard** — по-богато разнообразие на генерираните сайтове.
