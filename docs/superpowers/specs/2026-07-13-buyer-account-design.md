# Купувачески профил — дизайн (спец)

> **Дата:** 2026-07-13 · **Статус:** одобрен, чака имплементационен план.
> Голямата „самостоятелна" функция от `docs/remaining-roadmap.md`. Стъпва на готовия
> social login (Google) — [[social-login-feature]] в паметта, спец
> `2026-07-12-social-login-design.md`.

## Цел

Логнат купувач вижда историята на поръчките си, пази адресна книга за бърз checkout и
синхронизира любимите между устройства. Retention функция; social login смъкна цената ѝ.

## Обхват (среден) — решения на потребителя (2026-07-13)

- **Среден обхват:** „Моите поръчки" + адресна книга (с опция за куриерски офис) + любими
  (синхрон per-акаунт вместо localStorage). Ревюта/Q&A **остават както са** (anonymous /
  verified по телефон) — НЕ се обвързват с акаунт в тази итерация.
- **Роля чрез toggle** на вход/регистрация (купувач/продавач), различен UI. Роля = избор
  при регистрация, но **един акаунт може да расте** (купувач по-късно отваря магазин, същият
  Supabase user, без нов имейл).
- **Авто-обвързване на стари гост-поръчки по потвърден телефон** — потвърждение = минала
  поръчка със същия телефон + **клик** за свързване (без SMS; равностойно на verified
  ревютата, но купувачът контролира → GDPR-приятелско).
- **„Моите поръчки" е per-магазин** (в контекста на `/s/{slug}`) за първа итерация.
  Кросс-магазинно табло = бъдеще.

## Изрично ИЗВЪН обхвата (YAGNI)

- ❌ Мулти-user екипи (роли/покани per магазин) — **премахнати от пътната карта** (overkill).
- ❌ Кросс-магазинно табло на поръчките.
- ❌ SMS потвърждение на телефон (клик-свързване вместо това).
- ❌ Обвързване на ревюта/Q&A с акаунт.
- ❌ Wishlist споделяне, лоялни точки, купувачески купони.

## Архитектура

### Роли — производни, не твърди

`profiles` остава **една** таблица (споделена от търговци и купувачи; един Supabase user =
един profile ред). `shops.ownerId → profiles.id` вече съществува.

- **Истинската роля е производна:** имаш магазин (`shops.ownerId = you`) → търговец; иначе
  → купувач. Няма втори източник на истина.
- Toggle-ът на входа е само **UX насока** (redirect след вход) + козметика на екрана.
  Записва се като `profiles.preferredRole` (nullable, `'buyer'|'seller'`) — чисто redirect
  памет, **не гард, не заключва**.
- Гъвкавост: купувач натиска „Отвори магазин" от профила → сегашният create-shop поток със
  същия акаунт.

### Данен модел (`src/db/schema.ts`)

Всички нови колони nullable / нови таблици → **`db:push` е безопасен** (без загуба; аудитният
урок за версии спазен). Всички нови таблици с `.enableRLS()`.

**`profiles` — 2 нови незадължителни полета:**
```
preferredRole: text (nullable)        // 'buyer' | 'seller' — само redirect памет
phoneVerified: boolean default false  // true след клик-свързване на минали поръчки
```
(`fullName`, `phone`, `id`, timestamps вече ги има.)

**`orders` — нов `buyerId`:**
```
buyerId: uuid (nullable) → profiles.id (onDelete: set null)
```
- NULL = гост поръчка (както сега). Попълва се при checkout ако купувачът е логнат ИЛИ при
  клик-свързване по потвърден телефон.
- Индекс `orders_buyer_idx` on `(buyerId, createdAt)`.
- Поръчката пази `shopId` — купувачът чете само своите поръчки по `buyerId`; **тенант
  изолацията за търговеца не се чупи** (търговските заявки остават по `shopId`).

**Нова таблица `buyerAddresses` (адресна книга, покрива адрес-до-врата И куриерски офис):**
```
id, buyerId → profiles.id (onDelete cascade)
label            // „Вкъщи", „Работа" — по избор
receiverName, receiverPhone
city, address                       // адрес до врата
courierProvider (nullable)          // 'econt'|'speedy' ако е запазен ОФИС вместо адрес
courierOfficeId (nullable)
courierOfficeName (nullable)
isDefault: boolean
createdAt, updatedAt
index on (buyerId)
```
Един запазен адрес е „до врата" ИЛИ „до офис"; checkout autofill попълва каквото е приложимо.

**Нова таблица `buyerFavorites` (синхрон):**
```
id, buyerId → profiles.id (onDelete cascade)
productId → products.id (onDelete cascade)
createdAt
uniqueIndex on (buyerId, productId)   // без дубли
index on (buyerId)
```
Заместник на localStorage **за логнати**. Гост още ползва localStorage; при вход → merge.
Тенант бележка: `productId` пресича магазини (купувачът пазарува от много) — ОК, изолацията
важи за търговски заявки; купувачът чете само своите редове по `buyerId`.

### Заявки и мутации — изолация правило №1 за купувача

Нов wrapper по аналогия с `requireShop()` (`src/lib/auth.ts`):
```
requireBuyer() = auth (Supabase user) → profile → { user, profile }
```
Всяка купувачка заявка/мутация филтрира по `buyerId = profile.id`. Криво `buyerId` (чужди
поръчки/адреси/любими) = критичен бъг, същият клас като cross-tenant.

**Нови файлове:**
- `src/db/queries/buyer.ts` — `getBuyerOrders(buyerId)`, `getBuyerAddresses(buyerId)`,
  `getBuyerFavoriteIds(buyerId)`, `getBuyerFavoriteProducts(buyerId)`
- `src/actions/buyer.ts` — `saveAddress`, `deleteAddress`, `setDefaultAddress`,
  `updateBuyerProfile`, `toggleBuyerFavorite`, `mergeFavoritesOnLogin(localIds[])`,
  `linkGuestOrders()` (клик-свързване)
- `src/schemas/buyer.ts` — Zod (адрес, профил); реюз `parseBgPhone`, `sanitizeText`

### Авто-обвързване на гост-поръчки (по потвърден телефон)

- Без SMS (външна платена услуга, извън обхвата). Използваме наличния сигнал: ако купувачът
  има поне една гост-поръчка (`buyerId` NULL) с телефона на профила си → при регистрация/вход
  показваме „Намерихме N твои минали поръчки — свържи ги с акаунта?" → при **клик** `buyerId`
  се попълва на тези поръчки; `phoneVerified` става `true`.
- Равностойно на verified ревютата (телефон срещу реална поръчка), но изисква действие →
  не излага тихо чужди данни.
- Нови поръчки при логнат купувач → `buyerId` се пише директно в checkout.

### Merge на любими при вход

- `mergeFavoritesOnLogin(localIds)` — вливат се localStorage ID-тата в `buyerFavorites`
  (upsert, uniqueIndex пази от дубли), после клиентът чисти localStorage за този shopId.
  Извиква се веднъж след успешен вход.
- `favorites-storage.ts` получава „logged-in" режим: чете от сървъра (`buyerFavorites`),
  пише през `toggleBuyerFavorite`. `useSyncExternalStore` остава — сменя се само източникът.

## UI

### Storefront хедър (`src/components/storefront/header/`)
- Нова икона „Профил" (`<Icon name="user">`) до сърцето и количката.
- Гост → линк към вход; логнат → dropdown („Моите поръчки", „Адреси", „Любими", „Изход").
- Само `--sf-*` токени (storefront тема, НЕ „Пазарен ден"); един компонент за всички теми.

### Профилни страници под `/s/{slug}/account` (route група `(storefront)`)
- `/account` — табло: поздрав + бързи връзки + последна поръчка
- `/account/orders` — „Моите поръчки" (per-магазин): статус, „Поръчай пак" (реюз
  `reorder-button`), детайл по `publicToken`
- `/account/addresses` — адресна книга (CRUD през `<Drawer>`, не модал)
- Любими — **сегашната `/s/{slug}/favorites` остава единствената страница** (не дублираме
  под `/account`); от профилното меню „Любими" сочи натам. За логнати източникът става
  сървърен (`buyerFavorites`), за гости — localStorage (без промяна на URL-а).
- `/account/settings` — име/телефон/имейл + „Отвори магазин" (расте към продавач) +
  изтриване (реюз account-deletion)
- Навигация: новият `ui/Tabs` примитив (Фаза 1), `?tab=` в URL. Плитко, консистентно.

### Платформен вход (`/auth/login`, `/auth/register`) — toggle
- Сегментиран контрол горе: „Пазарувам" / „Продавам".
- Различен UI по избор (темови подпис през токени, mobile-first):
  - *Купувач*: „Влез, за да следиш поръчките си и любимите"
  - *Продавач*: „Създай своя магазин" (сегашният екран)
- Google бутонът работи за двете (готов); toggle-ът определя само `next` default.
- `AuthForm` получава `role` prop → сменя copy + `next` default.
- Redirect след вход: `preferredRole='seller'` ИЛИ има магазин → `/dashboard`; иначе →
  `/account` (или `next` ако е дошъл от магазин).

### Правила за състояния
- Loading/error/empty за всяко async view. **`loading.tsx` НЕ на CRUD страници с drawer
  форми** (адреси) — remount-ва при `router.refresh()` и затваря drawer-и.
- Форми в `<Drawer>` (fullscreen на мобилно), не модали; не се затварят при клик отвън.

### Контекст на профила (съзнателно опростяване)
Купувачът се логва в контекста на магазин (`/s/{slug}/account`), защото оттам пазарува.
Акаунтът е един (Supabase). В друг магазин вижда своя профил, но „Моите поръчки" показва
поръчките **от този магазин**. Кросс-магазинно табло = бъдеща итерация.

## Грешки, сигурност, тестове

**Грешки:** общи BG съобщения без вътрешни детайли. Rate limit на публичните купувачки
мутации (адрес, merge, link) през наличния `src/lib/rate-limit.ts`.

**Тестове (Vitest unit, TDD за чистата логика):**
- `requireBuyer()` — гард, отхвърля нелогнат
- Изолация (критичен): `getBuyerOrders/Addresses/Favorites` връщат само редове на този
  `buyerId` (cross-buyer теч)
- `mergeFavoritesOnLogin` — дедуп (uniqueIndex), празен вход, дубли
- `linkGuestOrders` — намира гост-поръчки по телефон, свързва само след клик, не пипа чужди
- Zod схеми (адрес, профил) — валидни/невалидни, `parseBgPhone`
- Redirect логика (seller/има магазин → dashboard; иначе → account)

**E2e (Playwright, само гейт):**
- Регистрация като купувач → toggle → профил
- Гост поръчка → регистрация със същия телефон → клик-свързване → вижда я в „Моите поръчки"
- Адрес CRUD; любим синхрон (localStorage → сървър при вход)
- `@gmail.com` алиаси; cookie банер през `addInitScript` (`frizmo-cookie-notice=1`)

## Обобщение на одобрените решения (2026-07-13)

1. Среден обхват: поръчки + адресна книга (с офис опция) + любими синхрон.
2. Toggle купувач/продавач на входа, различен UI; `preferredRole` = само redirect памет,
   истинската роля производна от „имаш ли магазин".
3. Авто-обвързване по потвърден телефон = минала поръчка + клик (без SMS).
4. Per-магазин „Моите поръчки".
5. Мулти-user екипи — премахнати от обхвата.
