# Frizmo Shops — MVP дизайн спецификация

**Дата:** 2026-07-02
**Статус:** Одобрена концепция, готова за имплементационен план
**Автор:** Емил + Claude (брейнсторминг сесия)

---

## 1. Какво е Frizmo Shops

SaaS платформа за българския пазар, чрез която малки бизнеси създават собствен онлайн магазин без технически познания. Комбинира две неща:

1. **Инструмент** за създаване и управление на онлайн магазин (админ панел на търговеца);
2. **Каталог/маркетплейс**, в който купувачите откриват всички активни магазини и продукти.

**Фаза:** валидация — има 2-3 реални запитвания. Оптимизираме за минимален месечен разход (0 € до реални клиенти) и възможност за растеж без пренаписване.

**Целеви ниши за v1:** малки производители / ръчна изработка, дрехи и мода, храни и местни продукти. Платформата остава използваема и за други типове бизнеси.

## 2. Взети решения (фиксирани)

| Въпрос | Решение |
|---|---|
| SaaS или маркетплейс | Комбинация: инструмент + каталог |
| Количка | Per-магазин; една поръчка = един магазин (без cross-store количка) |
| Профил на купувача | Не — гост checkout; профили за купувачи извън MVP |
| URL на магазин | Път в платформата: `/s/{slug}`; субдомейн/собствен домейн — по-късен етап |
| Ценови модел | 2 платени плана (Starter/Pro) + 14 дни trial с пълен Pro достъп, без карта |
| Приходи | Само абонаменти; без комисиона от продажби |
| Модерация | Без предварително одобрение; платформен админ може да скрие/блокира магазин |
| Тип продукти | Само физически; дигитални и услуги — извън MVP |
| Плащане с карта (купувач → търговец) | Извън MVP; методи: наложен платеж, банков превод, на място |
| Куриерски интеграции | Извън MVP; доставката е ръчни настройки на търговеца |

Извън MVP още: рейтинги/ревюта от реални купувачи, чат, фактури, имейл автоматизации, AI асистент, мобилно приложение, реклами в каталога, сложни промоции.

## 3. Технологичен стек

| Слой | Технология | Защо |
|---|---|---|
| Приложение | **Next.js 15 (App Router)** — фронтенд + бекенд в едно | SSR/ISR за SEO на публичните страници; един deploy; без отделен API сървър |
| Хостинг | **Vercel** (free tier → Pro при растеж) | 0 € старт, автоматични preview deployments |
| База данни | **Supabase Postgres** + **Drizzle ORM** | Релационен модел (продукти/варианти/поръчки/филтри), full-text търсене без външни услуги |
| Auth | **Supabase Auth** (търговци) | Вградено, без допълнителна услуга |
| Файлове | **Supabase Storage** (снимки) | Подписани URL-и, изолация per-магазин |
| Валидация | **Zod** — едни схеми за клиент и сървър | Познат модел от Frizmo |
| Стилове | **Tailwind CSS 4** с design tokens | Виж §7 |
| Абонаменти | **Stripe** (Checkout + webhooks) | Познат от Frizmo |
| Имейли | **Resend** | Free tier 100/ден — достатъчно за MVP |
| Блог | **MDX файлове** в `content/blog/` | 0 €, статично генериране, перфектно SEO |
| Тестове | **Vitest** (unit) + **Playwright** (e2e) | |

**Разходи:** MVP = 0 €/месец + домейн (~15-30 €/год). При ~20-50 магазина: ~45 €/месец (Supabase Pro + Vercel Pro).

**Валута:** EUR (България е в еврозоната от 01.01.2026). Всички цени се съхраняват като integer евроцентове.

## 4. Архитектура и URL структура

Едно Next.js приложение с четири „лица":

```
/                          Landing на платформата (маркетинг)
/blog, /blog/{slug}        Блог (MDX, SEO)
/pricing                   Цени

/shops                     Каталог с магазини (търсене, филтри)
/products                  Каталог с продукти от всички магазини

/s/{slug}                  Публичен магазин (SSR/ISR, SEO)
/s/{slug}/products         Продукти на магазина (+ категории, търсене в магазина)
/s/{slug}/p/{productSlug}  Страница на продукт
/s/{slug}/about            За нас
/s/{slug}/contact          Контакти
/s/{slug}/terms            Условия (доставка/връщане/поверителност)
/s/{slug}/cart             Количка
/s/{slug}/checkout         Гост checkout

/auth/...                  Регистрация / вход за търговци

/dashboard                 Админ панел на търговеца (client-rendered)
  /dashboard/store         Данни за магазина
  /dashboard/products      Продукти
  /dashboard/categories    Категории
  /dashboard/orders        Поръчки
  /dashboard/website       Персонализация „Уебсайт"
  /dashboard/shipping      Плащане и доставка
  /dashboard/billing       Абонамент
  /dashboard/settings      Настройки

/admin                     Платформен админ (само за собственика):
                           всички магазини, скриване/блокиране, базова статистика
```

**Рендериране:**
- Публични страници (landing, каталог, магазини, продукти, блог): SSR/ISR с кеш; инвалидация на кеша при промяна на продукт/магазин.
- Dashboard и /admin: client-side (SEO не е нужно).
- Количка: localStorage, отделна per-магазин.

**SEO инфраструктура от ден 1:** `sitemap.xml` (автоматично: магазини + продукти + статии), `robots.txt`, канонични URL-и, OpenGraph тагове, JSON-LD (`Product`, `Store`, `Article`).

## 5. Модел на данните (Postgres, Drizzle)

Multi-tenant модел: `shopId` е тенант ключът; всяка заявка на търговец се филтрира по него.

**Идентичност и магазини**
- `profiles` — 1:1 със Supabase Auth user: име, телефон.
- `shops` — `ownerId`, име, `slug` (уникален), описание, лого, адрес, телефон, имейл, работно време, соц. мрежи, основна бизнес категория, статус: `draft | published | suspended | blocked`.

**Продукти**
- `categories` — per-магазин; `parentId` за подкатегории (макс. 1 ниво в MVP); подредба.
- `products` — име, `slug`, описание, цена, промо цена (nullable), снимки (масив пътища в Storage), `categoryId`, статус `active | inactive`, наличност (nullable = „неограничена/не се следи"), подредба.
- `product_attributes` — гъвкави характеристики key/value per-продукт: („материя", „памук"), („произход", „Родопи"). Без твърда схема — решава „дрехи vs сирене vs тухли".
- `product_variants` — комбинация от опции с отделна цена/наличност: `{ options: {"размер":"M","цвят":"син"}, price, stock, sku? }`. Продукт без варианти няма записи тук.

**Поръчки**
- `orders` — `shopId`, пореден номер per-магазин (#0001…), клиент: име, телефон, имейл (nullable), адрес; метод на плащане, метод на доставка, цена на доставка, обща сума, бележка, статус: `new | confirmed | shipped | completed | cancelled`.
- `order_items` — snapshot: `productId`, вариант, име, количество, единична цена **към момента на поръчката**.

**Настройки**
- `shipping_methods` — per-магазин: тип (`courier | pickup | local`), име, цена, праг за безплатна доставка (nullable), активен.
- `payment_methods` — per-магазин: тип (`cod | bank_transfer | on_site`), детайли (IBAN за превод), активен.
- `site_settings` — 1:1 с магазин; JSON, валидиран срещу Zod схема `SiteSettings` (виж §6).

**Абонаменти**
- `subscriptions` — `shopId`, план `starter | pro`, Stripe customer/subscription id, статус lifecycle: `trial → active → grace → suspended → cancelled`, `trialEndsAt`, `graceEndsAt`, `paymentFailureCount`.

**Правила за всички таблици:** `id`, `createdAt`, `updatedAt`; `shopId` където е приложимо; пари като **integer евроцентове, валута EUR** (никога float); индекси на всички FK + търсени колони; full-text индекс (`tsvector` + `pg_trgm`) върху имена/описания на продукти и магазини.

## 6. Публичният магазин — пълноценен сайт

### 6.1 Страници
Начало (секции), Продукти (категории + търсене в магазина + breadcrumbs + свързани продукти на продуктовата страница), За нас, Контакти (форма + карта + работно време), Условия — **генерирани от шаблон** с данните на търговеца (доставка, връщане, поверителност; законово задължителни по ЗЗП).

### 6.2 Секционна система за началната страница
`site_settings.sections` е подреждаем масив от типизирани секции. Всяка секция = React компонент + Zod схема на настройките ѝ. Търговецът добавя/маха/пренарежда (dnd-kit) и попълва съдържание. Без свободен HTML от потребителя.

Каталог от секции в MVP:

| # | Секция | Настройки | План |
|---|---|---|---|
| 1 | Hero | 3 layout-а (пълна снимка / split / slideshow), заглавие, подзаглавие, CTA | Starter (1 layout) / Pro (всички) |
| 2 | Announcement bar | текст, линк | Pro |
| 3 | Избрани продукти | ръчен избор или auto (най-нови/промоции) | Starter |
| 4 | Категории grid | избор на категории, снимки | Starter |
| 5 | Промо банер | снимка, текст, бутон | Pro |
| 6 | Снимка + текст | редуващи се страни, свободен rich text („изберете нашето мляко, защото…") | Starter |
| 7 | Свободен текст | rich text (заглавия, списъци, bold; без HTML) | Starter |
| 8 | Отзиви | ръчно въведени: име, текст, снимка | Pro |
| 9 | Trust badges | избор от готов набор икони + текст | Pro |
| 10 | Галерия | снимки | Pro |
| 11 | FAQ акордеон | въпроси/отговори | Pro |
| 12 | Карта + контакти | адрес, работно време | Starter |
| 13 | Социални мрежи | линкове с икони | Starter |

**Глобални елементи:** Header (лого, навигация, търсене, количка; 2 варианта подредба), Footer (контакти, условия, соц. мрежи), тема + цветове.

### 6.3 Теми и стилове на магазина
- 3 готови теми в MVP: `classic`, `modern`, `warm` — набор от CSS variables (цветове, шрифтове, радиуси) + вариации в layout-а на секциите.
- Търговецът избира тема + primary/accent цвят (валидиран hex). Настройките се излива като CSS variables на `<html>` на магазина; компонентите на storefront-а четат **само** тези променливи.
- Starter: тема `classic`; Pro: всички теми.

### 6.4 Preview
Iframe с живия магазин + `postMessage` за настройките → мигновен преглед преди запис (патърнът от Frizmo Website Builder).

## 7. Дизайн система на платформата

- **Design tokens** в `src/styles/tokens.css` чрез Tailwind 4 `@theme`: цветове, отстояния, радиуси, типография, сенки. **Нула хардкоднати стойности в компоненти.** Смяна на бранд = 1 файл.
- **Компонентна библиотека** `src/components/ui/`: Button, Input, Select, Textarea, Checkbox, RadioGroup, Modal, SlidePanel (mobile bottom-sheet), Card, Badge, Table, Tabs, Accordion, EmptyState, Spinner, Toast (Sonner), ImageUploader, PriceTag, QuantityPicker, Pagination, ConfirmDialog.
- Правило: **всеки елемент, използван на 2+ места, е компонент**. Страниците не съдържат голи `<button>`/`<input>` с класове.
- Всеки компонент: типизирани props, състояния loading/disabled/error, 44px touch targets, ARIA атрибути.
- Мобилен дизайн по подразбиране (375px минимум) — публичният трафик е предимно мобилен.

## 9. Поръчков поток и edge cases

**Гост checkout (без профил):**
1. Количка per-магазин в localStorage.
2. Checkout на 1 страница: име, телефон, имейл (по избор), адрес; избор от активните методи за доставка/плащане на магазина.
3. Сървърът преизчислява всичко: активност на продукти, наличност, актуални цени (нулево доверие към клиентските данни).
4. Транзакция: създаване на поръчка + декремент на наличности + пореден номер per-магазин.
5. Потвърждение на екран + имейл до купувача (ако е дал) и до търговеца.

**Задължителни edge cases:**
- Изчерпан/деактивиран продукт в количката → ясно съобщение per ред, количката се актуализира.
- Променена цена → купувачът вижда актуалната преди потвърждение.
- Race за последна бройка → транзакция с проверка; вторият получава „изчерпано".
- Suspended/blocked магазин → страница „временно затворено", checkout блокиран.
- Спам → rate limiting per IP + honeypot поле (без CAPTCHA).
- Отказана поръчка → наличностите се възстановяват.

**Търговецът:** списък (филтри по статус) → детайл → смяна на статус `new → confirmed → shipped → completed / cancelled`; badge за нови поръчки в dashboard.

## 10. Абонаменти и лимити

| | Starter (~10 €/м) | Pro (~20 €/м) |
|---|---|---|
| Продукти | 50 | Неограничени |
| Категории | 1 ниво | + подкатегории |
| Теми | classic | Всички 3 |
| Секции | Базови (виж §6.2) | Всички |
| Промо цени | — | ✓ |
| Статистика | Базова (поръчки, приходи) | Пълна (по продукти, периоди) |
| Каталог | Стандартно | Приоритетно класиране |

Цените са параметри — финализират се преди launch.

- **Trial:** 14 дни пълен Pro, без карта. Lifecycle: `trial → active → grace (7 дни при неуспешно плащане) → suspended`.
- Лимитите се проверяват **на сървъра**: `checkPlanLimit(shopId, feature)` във всяка релевантна Server Action.
- Downgrade/suspend: данни не се трият — продукти над лимита стават неактивни; suspended магазин показва „временно затворено".
- Stripe Checkout + webhooks + **дневна reconciliation** (cron) като предпазна мрежа.

## 12. Качество, сигурност, тестове

**Сигурност (всяка мутация):** wrapper `authorizedAction()` = auth → собственост по `shopId` → Zod → санитизация → изпълнение. Публични endpoint-и (checkout, контактна форма): rate limit + Zod + санитизация. Подписани URL-и за качване; лимити за размер/тип на файлове. Без stack traces към клиента; структурирани логове. Строг TypeScript (`strict`, без `as any`), ESLint.

**Тестове:**
- Vitest: пари (integer стотинки), наличности, планови лимити, slug, Zod схеми, order транзакция.
- Playwright e2e: (1) регистрация → магазин → продукт → публикуване; (2) гост поръчка end-to-end; (3) обработка на поръчка от търговеца.
- Гейт преди deploy: lint + tests + build.

**Производителност:** ISR кеш + инвалидация, `next/image` за всички снимки, пагинация на всички списъци, DB индекси (§5), full-text търсене в Postgres.

**Работен процес:** Git от ден 1; `dev` бранч → preview, `main` → production (Vercel).

## 13. Onboarding на търговеца

Воден процес след регистрация (стъпки с прогрес, могат да се прескачат и довършват после):
1. Данни за магазина (име → автоматичен slug, категория, контакти).
2. Първи продукт (опростена форма).
3. Избор на тема + цветове.
4. Преглед → **Публикувай**.

Табът „Уебсайт" и публикуването се отключват след стъпки 1-2 (има какво да се показва). До публикуване магазинът е `draft` — вижда се само от собственика през preview линк.

## 14. Дефиниция на готово за MVP

MVP е завършен, когато и трите e2e пътя работят на production домейн:
1. Търговец се регистрира, създава магазин с продукти/категории, персонализира го и го публикува — изцяло сам, без помощ.
2. Купувач намира магазина през каталога или директен линк, разглежда, поръчва като гост и получава потвърждение; търговецът получава нотификация и обработва поръчката.
3. Trial изтича → търговецът се абонира през Stripe → лимитите на плана му се прилагат коректно.

Плюс: landing + блог (мин. 2 статии) онлайн, 3 демо магазина попълнени, всички тестове минават.

Без TODO секции, без „coming soon" — всяка включена функция е завършена.
