# Дизайн: Табове на претрупаните dashboard страници (Фаза 1)

**Дата:** 2026-07-12
**Статус:** одобрен дизайн, чака ревю на спеца
**Обхват:** чист презентационен UX рефактор — нула нова бизнес логика, нула промяна в заявки/actions/планова система.

## Контекст и цел

Пет dashboard страници смесват несвързани теми на един дълъг екран (одит 2026-07-12).
Целта: разделяме претрупаните страници на табове, за да е по-лесно за навигиране —
по образеца на вече съществуващия `/dashboard/website` builder (split-view с табове).

Този рефактор подготвя **чисти шевове** за Фаза 2 (режим на сложност): след като
секциите са в табове, режимът ще може да скрива цели табове, вместо да реже вътре в
дълга колона.

**Разделено от плановата система.** Табовете нямат нищо общо с планове или билинг.

## Ключови решения (взети с потребителя)

1. **Състояние на таба в URL** (`?tab=...`) — поддържа refresh, back бутон, споделяне
   на линк и deep-link от онбординг чеклиста. Невалиден/липсващ `?tab` → тихо пада на
   първия таб.
2. **Плитка навигация** (`history.replaceState`, без re-fetch) — смяната на таб е
   мигновена, не прави сървърна заявка, и не чупи отворени drawer-и.
3. **Всички панели монтирани наведнъж**, неактивните скрити с CSS (`hidden`) — form
   state, dirty-guard и drawer state оцеляват при смяна на таб (нищо не се re-mount-ва).
4. **Мобилно (375px): хоризонтален скрол** на лентата с табове (като iOS/Material tab
   bar); активният таб `scrollIntoView`.
5. **Продуктова форма: табове само в „Детайлно" режим.** „Бързо" остава непокътнат
   (3 карти в колона). Съществуващият Бързо/Детайлно тогъл се запазва.

## Архитектура

### Нов примитив: `src/components/ui/tabs.tsx`

Един reusable, презентационен client компонент, изнесен в `src/components/ui/index.ts`.
Използва се от всичките 5 страници. Не знае нищо за домейна — приема конфиг + деца.

**Публичен интерфейс:**

```tsx
export interface TabItem {
  key: string;      // стойност в ?tab= (напр. "payment")
  label: string;    // видим надпис (напр. "Плащане")
  marker?: boolean; // ако true, показва точка (•) — за таб с валидационна грешка
}

export interface TabsProps {
  tabs: TabItem[];
  children: React.ReactNode; // по един <TabPanel> на таб, в реда на tabs
  paramName?: string;        // име на query param, default "tab"
  ariaLabel: string;         // aria-label на tablist
}

export function Tabs(props: TabsProps): JSX.Element;

export interface TabPanelProps {
  tabKey: string;            // трябва да съвпада с TabItem.key
  children: React.ReactNode;
}
export function TabPanel(props: TabPanelProps): JSX.Element;
```

**Поведение:**

- `"use client"`. Чете активния таб от `window.location` (`URLSearchParams`) при mount и
  държи го в state. При клик: `setActive(key)` + `history.replaceState(null, "", url)` с
  обновен param (плитко — НЕ `router.push`, НЕ re-fetch).
- Ако `?tab` липсва или не съвпада с никой `TabItem.key` → активен е `tabs[0].key`.
- Лента: `role="tablist"` + `aria-label`; всеки бутон `role="tab"`, `aria-selected`,
  `id`, `aria-controls` сочещо към панела; клавиатура ←/→ мести фокуса и активния таб,
  Home/End за първи/последен.
- Панели: `<TabPanel>` рендерира `role="tabpanel"`, `aria-labelledby`, `hidden` когато не
  е активен. **Всички панели са в DOM винаги** (само CSS `hidden`).
- Стил (само токени): лента с долен hairline `border-surface-200`; активен таб =
  `text-ink-900` + долен `border-brand-500` (2px); неактивен = `text-ink-500`
  `hover:text-ink-700`. Touch target `h-11`. `marker` точка = малка `bg-danger-500`.
  Лента: `flex overflow-x-auto` + скрит скролбар помощен клас; активен таб
  `scrollIntoView({ inline: "nearest", block: "nearest" })` при смяна.
- Тъмна тема: идва автоматично от предефинираните токени (без `dark:` варианти).

**Тестване (Vitest, `src/components/ui/tabs.test.ts` или `.tsx`):**
- default таб при липсващ `?tab`
- default таб при невалиден `?tab=xxx`
- избран таб при валиден `?tab=payment`
- клик сменя активния панел + обновява URL (mock на history)
- клавиатура ←/→ мести активния таб
- aria: `role=tablist/tab/tabpanel`, `aria-selected`, `aria-controls`/`aria-labelledby`

### Страниците остават server components

Всяка страница зарежда данните си както сега (нула нови заявки), после подава готовите
секции като `<TabPanel>` деца на `<Tabs>`. Data-flow не се променя.

## Разпределение на табовете

Нищо не се маха — само се групира. Изброените „секции" са днешните компоненти/блокове.

### 1. `/dashboard/fulfillment` — „Плащане и доставка" (3 таба)

| Таб (key) | Label | Съдържа |
|---|---|---|
| `shipping` | Доставка | методи за доставка + зони + работно време (вложени както са) |
| `payment` | Плащане | методи за плащане (вкл. IBAN валидация) |
| `orders` | Поръчки и връщания | целият `OrderSettings` (подаръчни опции + срок за връщане) |

Днес: `FulfillmentManager` + `OrderSettings` една под друга. Множество независими
manager-и/форми → всеки таб пази своето самостоятелно, без общ бутон.

### 2. `/dashboard/subscribers` — „Абонати" (3 таба)

| Таб (key) | Label | Съдържа |
|---|---|---|
| `subscribers` | Абонати | списък + `SubscribersExport` + таблица реферали |
| `campaigns` | Кампании | `CampaignComposer` + история на изпратени |
| `growth` | Купони за растеж | `GrowthSettingsForm` (welcome + реферален купон) |

Множество независими форми → без общ бутон.

### 3. `/dashboard/analytics` — „Аналитика" (3 таба)

Период selector-ът (7/30/90 дни) стои **НАД табовете** — общ за всички.

| Таб (key) | Label | Съдържа |
|---|---|---|
| `overview` | Общ преглед | 4 метрики + `RevenueChart` (приходи по дни) + топ продукти |
| `breakdowns` | Разрези | източници (плащане/град/купони) + топ категории |
| `customers` | Клиенти | конверсия абонати→клиенти + повторни клиенти |

Само визуално групиране на вече изчислените данни (`getAnalyticsBreakdowns`).

### 4. `/dashboard/store` — „Магазин" (3 таба)

`ShopForm` е **едно `<form>`** с един бутон „Запази" и dirty-guard. Остава един form —
табовете групират полетата визуално вътре. Бутонът „Запази" стои **под табовете** (общ,
пази цялата форма). Изтриване на акаунт стои **извън табовете** (отделна секция под тях).

| Таб (key) | Label | Полета |
|---|---|---|
| `basic` | Основни | име, категория, описание, статус |
| `contacts` | Контакти | адрес, град, телефон, имейл, работно време |
| `social` | Социални | 5-те социални мрежи |

**Валидационна грешка в скрит таб** → съответният `TabItem.marker = true` (точка), за да
не се „изгуби" грешката. Формата вече показва field-level грешки от Zod.

### 5. Продуктова форма (`ProductForm`) — само в „Детайлно" (4 таба)

„Бързо" режим остава непокътнат (3 карти в колона). В „Детайлно" 9-те карти се групират.
Формата е **едно `<form>`** → бутон „Запази" под табовете; marker при грешка в скрит таб.

| Таб (key) | Label | Карти |
|---|---|---|
| `main` | Основно | Основни, Цена и наличност, Снимки, Характеристики |
| `logistics` | Логистика | Тегло и размер, Количество, Таблица с размери |
| `codes` | Кодове и SEO | Продуктови кодове (SKU/GTIN/марка/доставна цена), SEO |
| `variants` | Варианти | Опции и варианти, Промоция „купи повече" |

## Крайни случаи

- **Deep-link от онбординг чеклиста:** релевантните стъпки получават `?tab=` (напр.
  „Попълни плащане" → `/dashboard/fulfillment?tab=payment`). Дефинира се per-стъпка в плана.
- **Drawer + смяна на таб:** при смяна на таб затваряме отворени drawer-и на напуснатата
  секция (за да не виси drawer от скрит таб). Дефинира се per-страница в плана.
- **Скрол:** плитка навигация не скролва прозореца; само хоризонтален `scrollIntoView` на
  активния таб в лентата.
- **Един form през табове (store, product):** бутон „Запази" общ, под табовете; marker на
  таб с грешка.
- **Множество форми (fulfillment, subscribers):** всеки таб самостоятелен, без общ бутон.
